# WB Media Downloader

React-приложение: по артикулу товара Wildberries показывает все фотографии карточки в
модальном окне с выбором и скачиванием в zip, а также находит и скачивает видео карточки
(если оно есть) по отдельности в mp4 — каждое честным HLS→fMP4 ремуксом на клиенте.

## Запуск

```bash
npm install
npm run dev
```

Приложение обращается к API Wildberries напрямую из браузера. В большинстве браузеров это
упирается в CORS (см. «Ограничения» ниже) — для локальной проверки можно включить любое
CORS-unblock расширение (например, «Allow CORS»).

Другие команды: `npm run build`, `npm run test`, `npm run lint`, `npm run typecheck`.

## Подход к получению медиа и почему

- **Карточка товара** — `GET https://card.wb.ru/cards/v4/detail?appType=1&curr=rub&dest=-1257786&nm={артикул}`.
  Технические подсказки задания указывают на
  `https://www.wildberries.ru/__internal/card/cards/v4/detail` — этот эндпоинт живой, но
  **блокирует кросс-доменные запросы** (403, подтверждено и с сервера, и из браузера с
  включённым CORS-расширением): он рассчитан на same-origin вызовы с самого
  wildberries.ru, а не на сторонние SPA. `card.wb.ru` — публичное зеркало того же API,
  реально отвечает 200 с браузера/сервера и используется вместо него.
- **CDN-хосты** — `GET https://cdn.wbbasket.ru/api/v3/upstreams`. Диапазоны `vol_range_from/to`
  для сопоставления `vol` → хост лежат в `origin.<route_map>`, а не в `recommend.<route_map>`
  (в `recommend` у хостов вообще нет `vol_range_*`, только имя хоста — это отдельная,
  round-robin/mod-схема, не подходящая для формулы из подсказки).
- **Фото** — по формуле из [гиста из задания](https://gist.github.com/sheldhur/50420d0b85b2c80bc02cf96550b50625):
  `vol = floor(nm / 1e5)`, `part = floor(nm / 1e3)`, хост — из `origin.mediabasket_route_map`.
  URL: `https://{host}/vol{vol}/part{part}/{nm}/images/big/{index}.webp`.
- **Видео карточки** — самая сложная часть задания; ниже раздел
  [«Сложности с видео и как решили»](#сложности-с-видео-и-как-решили) разбирает её подробно.
  Коротко: WB отдаёт видео только как HLS (плейлист `.m3u8` + сегменты `.ts`), никогда
  прямым mp4-файлом — это подтверждено вживую, а не предположение. Приложение собирает
  **все** источники видео, которые смогло найти (классическая схема + все готовые видео
  из отзывов), показывает по плитке ▶ на каждое в сетке фото и честно ремуксит HLS в
  fMP4 на клиенте через `mux.js` при скачивании любого из них.
- **Zip** — фото собираются на клиенте через `JSZip`: каждое выбранное фото фетчится как
  blob и добавляется в архив, упавшие URL собираются в список и не блокируют скачивание
  остальных.
- **Почему без бэкенда** — задание описывает «простое React приложение»; клиентский
  вызов WB API/CDN — самый прямой путь под это ограничение. CORS-издержки этого решения
  документированы ниже как явный, осознанный компромисс, а не забытая деталь.

## Сложности с видео и как решили

Это оказалась самая трудоёмкая часть задания — по факту WB video устроено не как одна
система, а как **три независимые**, и большая часть времени ушла на то, чтобы вживую (через
DevTools реальной страницы товара, прямые запросы к CDN, а не чтение документации, которой
просто нет) понять, какая из них вообще достижима без бэкенда.

**1. «Классическая» схема из гиста задания.** `vol = nm % 144`, `part = floor(nm / 1e4)`,
хост — из `origin.videonme_route_map` в `/api/v3/upstreams`. Путь `.../hls/{quality}/{name}`.
Первая версия реализации пробовала скачать `index.mp4` по этому пути — наивно предполагая,
что рядом с HLS может лежать прогрессивный mp4. На практике это никогда не срабатывало:
проверка вживую показала, что WB отдаёт только реальный HLS (`index.m3u8` + `.ts`-сегменты),
никакого mp4 рядом нет. Исправили: проверка идёт на честный `index.m3u8`, а не на угаданное
имя файла.

**2. Тестовый артикул из задания (`604174866`) не имеет видео по классической схеме
вообще.** Проверено переборно: HEAD/GET по всем комбинациям качество×{mp4, m3u8} и по
полутора десяткам соседних артикулов той же партии — везде 404. При этом на странице товара
видео в карусели медиа видно — значит, оно в какой-то другой системе.

**3. «Wibes»** — короткие вертикальные ролики (отдельный раздел сайта, что-то вроде
stories/reels), тоже показываются в общей карусели медиа карточки. У них свой route map
(`origin.wibesbasket_route_map`, `"method": "uuid"`, хосты `wibes-01..08.wbbasket.ru`), и
реальные `.m3u8` по нему существуют (проверено вручную по ссылке, найденной в DOM
реальной страницы). Проблема — `{uuid}` этого ролика **нигде не встречается ни в одном
публичном JSON-ответе**: ни `card.wb.ru`, ни `/api/v3/upstreams`, ни (как выяснилось
следующим пунктом) `feedbacks1.wb.ru` — проверено point-in-time поиском по телам всех
ответов сети через встроенный поиск DevTools Network. Сама страница `wildberries.ru`
оказалась чистым CSR-шеллом без embedded state (исходный HTML — пустой `<div id="app">` +
подключение JS-бандлов, никакого `window.__INITIAL_STATE__`), то есть этот uuid не
«зашит» в разметку — он приходит через какой-то ещё не перехваченный запрос. Для
продакшена сюда нужен либо официальный/недокументированный API, отдающий этот uuid, либо
headless-браузер, рендерящий страницу и вытаскивающий его из DOM/сети после полной загрузки
SPA — существенно тяжелее формата тестового задания, и это тот самый случай «упёрлись на
2+ часа», который по условиям задания честнее задокументировать как ограничение, чем
продолжать реверс-инжинирить.

**4. Найдена рабочая альтернатива — видео из отзывов покупателей.** Тот же ролик в
карусели медиа страницы товара может быть customer-review-видео, а не «родным» видео
карточки — и оно достижимо. Публичный, CORS-открытый эндпоинт
`GET https://feedbacks1.wb.ru/feedbacks/v1/{root}` (ключ — `root` карточки, не `nm`;
`root` уже приходит в ответе `card.wb.ru`) отдаёт отзывы, часть которых содержит
`video: { id: "{shard}/{uuid}", isReady }`. `shard` — 1-based порядковый номер хоста в
`origin.videofeedback_uuid_route_map` из `/api/v3/upstreams`; итоговый URL —
`https://{host}/{uuid}/index.m3u8`. Проверено вживую на артикуле `604174866`: реальный
отзыв к нему содержит рабочее видео по этой схеме, хотя у самой карточки нет видео по
классической схеме — то есть без этой находки видео для примера из задания не скачалось
бы вообще никогда.

**5. Оба рабочих источника (классика и отзывы) — это HLS, а не mp4.** Честное скачивание
«видео в формате mp4» требует не просто скачать плейлист, а распарсить его, скачать все
`.ts`-сегменты и **реально смуксить** их в fMP4 (ISO BMFF `ftyp/moov/moof/mdat`). Простое
переименование склеенных `.ts` в `.mp4` дало бы файл, который многие плееры всё равно бы
проиграли (сниффинг по содержимому), но это была бы подмена, а не честный mp4-контейнер.
Вместо этого `shared/lib/hls` использует `mux.js` (тот же ремуксер, что внутри `hls.js`/
video.js) — лёгкий, чистый JS, без тяжёлого `ffmpeg.wasm`. Результат проверен не только
тестами (замоканный `Transmuxer`), но и отдельным end-to-end прогоном на реальных данных:
скачанный и смукшенный файл имеет корректную box-структуру `ftyp → moov → moof → mdat →
moof → mdat` (проверено через `mux.js`' собственный `mp4.tools.inspect`) — то есть валидный
fMP4, не поддельный.

**6. UI: почему плитки по одному видео, а не чекбоксы + общий zip, как у фото.** На
реальной странице товара у карточки может быть несколько видео одновременно (в этом
примере у одних только отзывов `feedbackCountWithVideo: 26`). Сознательно решили **не**
делать по аналогии с фото (чекбоксы + один zip-архив), а показывать по отдельной плитке ▶
на каждое найденное видео, каждая скачивается по клику независимо:
- В самом задании кнопка «Скачать видео» описана в единственном числе и без слова
  «архив» («видео карточки… в mp4»), в отличие от фото, где явно сказано «в zip архиве» —
  архивация видео не вытекает из ТЗ.
- Технически каждое видео — это не просто файл, а полноценный HLS→fMP4 ремукс на
  клиенте; выполнять несколько таких ремуксов подряд/параллельно ради одного zip-клика
  ощутимо тяжелее по CPU/памяти в основном потоке браузера, чем независимые скачивания
  по требованию.
- Кнопка «Скачать видео» в футере модалки при этом сохранена и скачивает первое найденное
  видео — она отвечает буквальной формулировке задания; плитки — это способ добраться до
  всех остальных, если их несколько.

## Что доделали бы для продакшена

1. **Wibes.** См. пункт 3 выше — нужен официальный API с uuid контента либо
   headless-рендер страницы. Самая тяжёлая часть, сознательно оставлена не сделанной.
2. **Фото.** Собственный backend-прокси: снимает CORS на постоянной основе (не полагаясь
   на расширения браузера у пользователя), даёт возможность батчить/кешировать запросы к
   WB CDN и скрывает детали CDN-маршрутизации от клиента.
3. **Видео.** Вынести `downloadHlsAsMp4` в Web Worker — сейчас ремукс идёт в основном
   потоке и может на короткое время подвесить UI на длинных роликах.
4. Если видео из отзывов всё же нужно архивировать zip'ом (например, по явному запросу
   продукта) — переиспользовать `buildZipFromImages`-паттерн, но запускать ремуксы
   строго последовательно, а не параллельно, чтобы не держать в памяти сразу несколько
   декодированных видео.

## Ограничения

- **CORS на `card.wb.ru`/CDN фото-хостах.** Не отдают `Access-Control-Allow-Origin` для
  сторонних origin. В деве это обходится браузерным расширением; для продакшена нужен
  собственный прокси (см. выше). Без обхода CORS приложение показывает понятную ошибку
  под инпутом артикула вместо падения. Важно: `cdn.wbbasket.ru`, `feedbacks1.wb.ru` и
  видео-CDN хосты (`videonme-basket-*`, `videofeedback*`) — **CORS открыт** (`Access-Control-Allow-Origin: *`,
  подтверждено заголовками), это не общее свойство всех WB-хостов, а особенность именно
  `card.wb.ru`.
- **`__internal` эндпоинт из задания недоступен извне.** Подтверждено 403 и с сервера, и
  из браузера — реализация использует публичное зеркало `card.wb.ru` (см. «Подход» выше).
  Важно: сам этот эндпоинт тоже не содержит никаких полей о видео (проверено — идентичен
  по составу `card.wb.ru`), так что дело не в блокировке, а в том, что card-API в принципе
  не знает о видео из отзывов/wibes.
- **Нет надёжного флага «есть видео».** Ни один из известных публичных эндпоинтов не
  говорит прямо, есть ли у карточки видео и какого оно типа — приложение узнаёт это,
  пробуя оба источника (классический, затем отзывы) по очереди.
- **Wibes-видео недостижимо** без headless-браузера/reverse-engineering серверного
  рендера — см. «Сложности с видео и как решили», пункт 3. Ролик в карусели на странице
  может быть визуально виден, но плитки в приложении для него честно не появится.
- Нет ретраев сетевых запросов и пагинации/лимита на количество фото в архиве.
- Ремукс HLS→fMP4 идёт полностью в основном потоке браузера (без Web Worker) — см.
  «Что доделали бы для продакшена».
