# WB Media Downloader

React-приложение: по артикулу товара Wildberries показывает все фотографии карточки в
модальном окне с выбором и скачиванием в zip, а также скачивает видео карточки (если оно
доступно) в mp4.

## Запуск

```bash
npm install
npm run dev
```

Приложение обращается к API Wildberries напрямую из браузера. В большинстве браузеров это
упирается в CORS (см. «Ограничения» ниже) — для локальной проверки можно включить любое
CORS-unblock расширение (например, «Allow CORS»).

Другие команды: `npm run build`, `npm run test`, `npm run lint`, `npm run typecheck`.

## Подход к получению медиа и почему

- **Карточка товара** — `GET https://card.wb.ru/cards/v4/detail?appType=1&curr=rub&dest=-1257786&nm={артикул}`.
  Технические подсказки задания указывают на
  `https://www.wildberries.ru/__internal/card/cards/v4/detail` — этот эндпоинт живой, но
  **блокирует кросс-доменные запросы** (403, подтверждено и с сервера, и из браузера с
  включённым CORS-расширением): он рассчитан на same-origin вызовы с самого
  wildberries.ru, а не на сторонние SPA. `card.wb.ru` — публичное зеркало того же API,
  реально отвечает 200 с браузера/сервера и используется вместо него.
- **CDN-хосты** — `GET https://cdn.wbbasket.ru/api/v3/upstreams`. Диапазоны `vol_range_from/to`
  для сопоставления `vol` → хост лежат в `origin.<route_map>`, а не в `recommend.<route_map>`
  (в `recommend` у хостов вообще нет `vol_range_*`, только имя хоста — это отдельная,
  round-robin/mod-схема, не подходящая для формулы из подсказки).
- **Фото** — по формуле из [гиста из задания](https://gist.github.com/sheldhur/50420d0b85b2c80bc02cf96550b50625):
  `vol = floor(nm / 1e5)`, `part = floor(nm / 1e3)`, хост — из `origin.mediabasket_route_map`.
  URL: `https://{host}/vol{vol}/part{part}/{nm}/images/big/{index}.webp`.
- **Видео (классическая схема из задания)** — та же идея, но `vol = nm % 144`,
  `part = floor(nm / 1e4)`, хост — из `origin.videonme_route_map` (**другой** набор хостов,
  не тот, что у фото). URL: `https://{host}/vol{vol}/part{part}/{nm}/hls/{quality}/{name}`.
  Наличие видео у карточки WB нигде явно не флагует (см. ограничения), поэтому вместо чтения
  несуществующего поля из ответа карточки приложение **пробует** скачать видео и по факту
  определяет, есть оно или нет (HEAD-проверка по качествам из `VIDEO_QUALITIES`, высокое
  качество первым).
- **Zip** — фото собираются на клиенте через `JSZip`: каждое выбранное фото фетчится как
  blob и добавляется в архив, упавшие URL собираются в список и не блокируют скачивание
  остальных.
- **Почему без бэкенда** — задание описывает «простое React приложение»; клиентский
  вызов WB API/CDN — самый прямой путь под это ограничение. CORS-издержки этого решения
  документированы ниже как явный, осознанный компромисс, а не забытая деталь.

## Что доделали бы для продакшена (особенно по видео)

Видео на Wildberries — это оказалась не одна, а **две разные системы**, обнаруженные при
живой проверке (не по документации, а через DevTools реальной страницы товара):

1. **Классическая схема** (описана в гисте из задания, реализована в этом проекте):
   `vol = nm % 144` → `videonme-basket-XX.wbbasket.ru` → `.../hls/{quality}/{name}`. Путь —
   это HLS (плейлист + сегменты), а не прямой mp4-файл. Честное скачивание «видео в mp4»
   из HLS требует муксинга сегментов в браузере (например, через `ffmpeg.wasm` — тяжёлый
   wasm-бандл) либо серверного транскод-прокси. В этом проекте видео пробуется скачать
   напрямую по HLS-пути с именем `index.mp4` — если WB когда-либо отдаёт прогрессивный mp4
   рядом с HLS, это сработает; если нет — приложение честно сообщает, что видео недоступно.
2. **«Wibes»** — обнаруженная в процессе живого тестирования отдельная система
   видео-контента (короткие ролики, что-то вроде stories/reels). У неё свой route map в
   `/api/v3/upstreams` (`origin.wibesbasket_route_map`, `"method": "uuid"`, хосты
   `wibes-01..08.wbbasket.ru`), и реальные `.m3u8`-манифесты по ней существуют (проверено
   вручную: `https://wibes-04.wbbasket.ru/{uuid}/{name}/index.m3u8`). Проблема — `{uuid}`
   не выводится из артикула никакой формулой и не встречается ни в ответе `card.wb.ru`, ни
   в `/api/v3/upstreams`; похоже, что он приходит вместе с серверным рендером страницы
   товара (встроенным в HTML, а не отдельным публичным API-вызовом) — то есть недоступен
   стороннему клиенту без реверс-инжиниринга самого HTML-рендера wildberries.ru, который
   сам по себе защищён от программного доступа (см. ограничения). Для продакшена сюда
   нужен либо официальный/недокументированный API поставщика данных о карточке,
   возвращающий этот UUID, либо серверный компонент, который сам рендерит/парсит страницу
   товара headless-браузером — оба варианта существенно тяжелее, чем формат тестового
   задания.
3. Для фото — в продакшене стоило бы завести собственный backend-прокси: снимает CORS
   на постоянной основе (не полагаясь на расширения браузера у пользователя), даёт
   возможность батчить/кешировать запросы к WB CDN и скрывает детали CDN-маршрутизации от
   клиента.

## Ограничения

- **CORS.** WB API (`card.wb.ru`, CDN-хосты) не отдаёт `Access-Control-Allow-Origin` для
  сторонних origin. В деве это обходится браузерным расширением; для продакшена нужен
  собственный прокси (см. выше). Без обхода CORS приложение показывает понятную ошибку
  под инпутом артикула вместо падения.
- **`__internal` эндпоинт из задания недоступен извне.** Подтверждено 403 и с сервера, и
  из браузера — реализация использует публичное зеркало `card.wb.ru` (см. «Подход» выше).
- **Нет надёжного флага «есть видео».** Ни `card.wb.ru`, ни `/api/v3/upstreams` не говорят,
  есть ли у карточки видео и в каком формате (классика или wibes) — приложение узнаёт это
  только пробуя скачать по классической схеме.
- **Видео работает только для классической схемы.** Если у карточки видео в формате
  «wibes» (как у тестового артикула 604174866 из задания) — кнопка «Скачать видео» честно
  останется недоступна, т.к. эта схема не была доступна для реализации в рамках задания
  (см. пункт 2 выше).
- **Даже классическая HLS-схема не гарантирует прогрессивный mp4** на каждом качестве —
  приложение пробует несколько качеств и сообщает, если ни одно не подошло.
- Нет ретраев сетевых запросов и пагинации/лимита на количество фото в архиве.
