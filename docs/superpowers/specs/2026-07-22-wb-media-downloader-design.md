# WB Media Downloader — Design

Date: 2026-07-22

## Purpose

Test/учебное задание: простое React-приложение, которое по артикулу товара Wildberries
показывает все фотографии карточки в модальном окне с возможностью выбора и скачивания
выбранных фото в zip-архиве, а также скачивания видео карточки (если есть) в mp4.
Внешний вид не важен — оценивается подход к получению медиа, архитектура и код.

## User flow

1. Пользователь вводит артикул в инпут и нажимает «Скачать фото».
2. Приложение запрашивает карточку товара по артикулу и открывает модальное окно.
3. В модалке показаны все фото карточки; по умолчанию все выбраны чекбоксами.
4. Кнопка «Скачать фото» — собирает выбранные фото в zip и скачивает архив.
5. Кнопка «Скачать видео» — скачивает видео карточки (если оно есть) в mp4 в высоком качестве.

## Approach

Client-only SPA (без бэкенда). Браузер напрямую обращается к WB API (получение карточки)
и к CDN (upstreams/hosts + сами файлы фото/видео). Если прямые запросы упрутся в CORS —
это документируется в README как известное ограничение с планом Б (proxy), без усложнения
кода в рамках тестового задания.

Ссылки-подсказки из задания (WB card API, gist с функциями генерации медиа-URL, CDN
upstreams) используются как референс для реализации `shared/lib/media-url`, а не как
предмет отдельного реверс-инжиниринга.

## Architecture (Feature-Sliced Design)

```
src/
  app/            — точка входа: Redux store (RTK Query), MUI ThemeProvider, глобальные стили
  pages/
    home/         — единственная страница: форма поиска + модалка
  features/
    search-card/       — инпут артикула + кнопка «Скачать фото»: триггерит запрос карточки
                          и открытие модалки
    media-selection/    — модалка: сетка фото с чекбоксами (по умолчанию все выбраны),
                          кнопки «Скачать фото» / «Скачать видео»
  entities/
    product-card/   — RTK Query api-slice (карточка + upstreams/hosts), типы, доменная модель
  shared/
    api/            — базовый RTK Query baseQuery, обработка сетевых/CORS-ошибок
    lib/
      media-url/    — чистые функции generateImageUrl / generateVideoUrl (логика из gist)
      download/     — buildZipFromImages (JSZip), downloadFile (blob → <a download>)
    ui/             — обёртка MUI-темы, общие мелкие компоненты
    config/         — константы (WB API URL, размеры фото и т.п.)
```

Слой `widgets` сознательно не используется — для одной модалки в небольшом приложении он
избыточен; вся логика модалки живёт в `features/media-selection`.

Выбор фото в модалке — локальный React state внутри фичи, не Redux (это UI-state, а не
серверное состояние).

## Data flow

1. `search-card` вызывает `useGetCardQuery(articleId)` (RTK Query) → карточка + upstreams.
2. `entities/product-card` через `shared/lib/media-url` строит массив URL фото и URL видео
   (если у карточки есть видео).
3. `media-selection` рендерит фото и хранит выбор в локальном state.
4. «Скачать фото» → `shared/lib/download.buildZipFromImages(selectedUrls)`: фетчит каждое
   фото как blob, кладёт в JSZip, отдаёт архив через `downloadFile`.
5. «Скачать видео» → сначала пробуем прямой mp4-URL по паттерну CDN (по аналогии с фото).
   Если WB отдаёт только HLS (m3u8 + сегменты) и прямого mp4 нет — это документируется как
   ограничение теста и раздел «что доделать для продакшена» (HLS→mp4 муксинг через
   ffmpeg.wasm или собственный бэкенд-прокси).

## Error handling

- Невалидный/несуществующий артикул → инлайн-ошибка под инпутом, модалка не открывается.
- CORS/сетевая ошибка при получении карточки → понятное сообщение пользователю
  («возможно, нужен прокси»), приложение не падает.
- Ошибка загрузки отдельного фото при сборке zip → фото пропускается, остальные всё равно
  упаковываются, в конце — короткое уведомление о пропущенных файлах.

## Tooling

- Vite + React + TypeScript (strict mode)
- RTK Query (Redux Toolkit) для server state
- MUI для UI-компонентов
- ESLint + Prettier + Vitest
- JSZip для сборки архивов
- npm как пакетный менеджер
- Роутинг не используется (одна страница)

## Testing scope

Юнит-тесты (Vitest) на чистые функции: `shared/lib/media-url` (по примерам из gist) и
`shared/lib/download` (сборка zip из мок-данных). UI/интеграционные тесты — не приоритет
(во view задания явно указано, что внешний вид не важен); реальную проверку в браузере
выполняет tester-агент через Playwright MCP.

## Multi-agent setup

`.claude/agents/`:
- `orchestrator.md` — ведёт весь цикл: разбивает задачу на шаги, делегирует реализацию
  `developer`-агенту, проверку — `tester`-агенту. Tools: `Agent` (+ планирование), без
  `Write`/`Edit`/`Bash` — сам код не пишет.
- `developer.md` — реализует FSD-слайсы, RTK Query, MUI-компоненты по шагам от orchestrator,
  следует code-style из корневого CLAUDE.md. Tools: `Read, Write, Edit, Bash, Grep, Glob`.
- `tester.md` — прогоняет `vitest`/lint/typecheck, через Playwright MCP проверяет реальный
  флоу в браузере (ввод артикула → модалка → выбор фото → скачивание zip/видео).
  Tools: `Bash, Read, Grep` + Playwright MCP.

Модель взаимодействия: orchestrator ведёт весь цикл разработки задачи, делегируя шаги
developer/tester через Agent tool.

## Project governance

- Корневой `CLAUDE.md`: команды (dev/build/lint/test/typecheck), правила FSD (направление
  импортов `app → pages → features → entities → shared`, обязательный public API через
  `index.ts` на каждом слайсе, запрет импортов «вбок» между слайсами одного слоя),
  код-стайл (TS strict, без `any`, конвенции именования RTK Query api-слайсов и файлов),
  краткое описание ролей агентов и когда их вызывать.
- `.mcp.json` в корне проекта: `context7` (доки библиотек), Playwright MCP (браузерная
  проверка для tester-агента), GitHub MCP (работа с issues/PR). Токены — через переменные
  окружения (например `${GITHUB_TOKEN}`), не хардкодятся в конфиге.

## Deliverables

- Git-репозиторий с историей коммитов по этапам (не один коммит «final»):
  scaffold → api-слой → media-url → форма поиска → модалка+выбор → zip-скачивание →
  видео-скачивание → README.
- `README.md` с разделами: подход к получению медиа и почему; что доделали бы для
  продакшена (особенно по видео); известные ограничения.
- Короткая демо-запись (gif/видео, 20–40 сек) — вне скоупа автоматической реализации,
  готовится вручную после того, как приложение работает.
